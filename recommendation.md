### Production deployment recommendations

To harden this crawler and searcher for a real production environment, the first step would be to separate concerns into independent services: a crawl scheduler, a fleet of crawler workers, an indexing pipeline, a search API, and a UI/API gateway. The current single-process Python implementation can evolve into a **worker template**: wrap the crawler loop into a process that pulls crawl tasks from a durable queue (for example, Kafka, RabbitMQ, or a managed cloud queue), and writes fetched pages to a durable storage and indexing service. The inverted index should be moved from in-memory Python data structures into a dedicated search engine (such as OpenSearch/Elasticsearch or a managed search service), which will provide scalable indexing, query-time ranking, and operational tooling like shard rebalancing and snapshots. For persistence, replace the local SQLite database with a replicated database (PostgreSQL or similar) to hold crawl jobs, page metadata, and rate-limit state, and back it with periodic backups and observability.

On the operational side, the system should adopt **strong politeness and safety controls** before crawling the public internet. That includes honoring `robots.txt`, enforcing per-domain and global rate limits, respecting crawl budgets per domain, and providing a “kill switch” for misbehaving jobs. The services should run inside containers orchestrated by Kubernetes or another orchestrator, with metrics and tracing exported to Prometheus/Grafana or an equivalent stack. Authentication and authorization should be added around the APIs, and audit logging enabled for search queries if required. Finally, the ranking model could be iteratively improved from the current term-frequency heuristic to BM25 or vector-based semantic search, while still keeping an interpretable “debug view” so engineers and students can understand why particular URLs are returned for a query.
